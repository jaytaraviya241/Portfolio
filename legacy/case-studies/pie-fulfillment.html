🧩 The Problem

The client runs a regional pie business delivering and offering pickup across specific locations. Their existing Shopify store had no enforcement of delivery zones — customers could place orders to zip codes the business couldn't serve, causing fulfillment failures and manual order cancellations every week.

They also needed a bundle builder for mixed pie orders, with different pricing thresholds for pickup vs. delivery, and a UI that conditionally showed delivery windows based on the customer's zip code. Every solution they found required expensive third-party apps with recurring monthly costs and additional performance overhead.

The core challenge: business logic that belongs in code was scattered across spreadsheets, staff notes, and manual order checks. We needed a single source of truth that the merchant could update without a developer.

📋 Scope

  • Custom OS 2.0 bundle builder section with schema settings
  • Zip code delivery zone checker (client-side + server-side validation)
  • Conditional fulfillment UI (pickup vs. delivery options)
  • Cart attributes and line item properties for order context
  • Shopify Function for cart_checkout_validation enforcement
  • PIE_CONFIG global metafield system as single source of truth
  • Dev theme workflow on live store (never on published theme)
  • PieRules.evaluate() shared rules engine
  • Testing matrix covering all business rule scenarios

⚙️ Technical Build

The architecture centers on a PIE_CONFIG global object sourced from Shopify shop metafields. This single source of truth powers both the client-side UI and the server-side Shopify Function — meaning the merchant can update delivery zones, pricing thresholds, or pickup locations from the Shopify admin without touching code.

The PieRules.evaluate() rules engine is a shared JavaScript module used across the bundle builder, cart, and checkout validation. It evaluates any zip code against the delivery zones list and returns the appropriate fulfillment options and UI state.

pie-rules.js JavaScript 
/**
 * PieRules — Shared rules engine
 * Single source of truth for all fulfillment logic
 */
class PieRules {
  constructor(config) {
    this.config = config || window.PIE_CONFIG;
  }

  evaluate(zipCode) {
    const zones = this.config.deliveryZones;
    const inZone = zones.includes(zipCode.trim());

    return {
      canDeliver: inZone,
      canPickup: true,
      fulfillmentOptions: inZone
        ? ['delivery', 'pickup']
        : ['pickup'],
      deliveryWindows: inZone
        ? this.config.deliveryWindows
        : []
    };
  }

  validateCart(cartItems, fulfillmentType) {
    const total = cartItems.reduce((sum, item) =>
      sum + (item.price * item.quantity), 0);
    const min = fulfillmentType === 'delivery'
      ? this.config.deliveryThreshold
      : this.config.pickupThreshold;

    return { valid: total >= min, shortfall: Math.max(0, min - total) };
  }
}

export default PieRules;

The Shopify Function for cart checkout validation runs server-side, providing a safety net that client-side JavaScript alone can't offer. Even if a customer bypasses the UI, the Function prevents invalid orders from completing.

cart_checkout_validation/run.js Shopify Function 
// Shopify Function — cart_checkout_validation
// Enforces fulfillment rules at the checkout level
export function run(input) {
  const errors = [];
  const attrs = input.cart.attribute;
  const zipCode = attrs?.find(a => a.key === 'delivery_zip')?.value;
  const type = attrs?.find(a => a.key === 'fulfillment_type')?.value;

  if (type === 'delivery' && !isValidZip(zipCode)) {
    errors.push({
      localizedMessage: "Sorry, we don't deliver to that zip code.",
      target: "cart"
    });
  }

  return { errors: errors };
}

💭 Reflection

This project reinforced several principles I now hold as non-negotiable in Shopify development:

  • Never develop directly on a live theme. Using an unpublished dev theme protected the client's live store throughout the entire build.
  • Centralize business rules. The PIE_CONFIG metafield system meant the rules were in one place — not scattered across templates, JS files, and app settings.
  • Client-side validation is UX. Server-side validation is business logic. The Shopify Function was non-negotiable — it's the only thing that makes the system actually reliable.
  • App-free isn't just about performance. It's about giving merchants a store they can operate and maintain. Every app is a dependency and a monthly cost.
